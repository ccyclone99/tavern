# Tavern Future System Roadmap SPEC

本文档把现有多个 SPEC 归拢成未来整体方案。目标不是再堆功能入口，而是让 Tavern 稳定成为“一个输入框驱动的叙事 RPG 引擎”。

## 1. 当前基线

已具备：

- 单输入框自然输入与 `IntentRouter` 分流。
- 行动预览、风险、统一 DC、被动检定。
- 装备、物品、商店、休息、升级、属性点。
- 探索奖励、线索账本、挑战、时钟、失败状态。
- 通关/失败记录与协议标记清理。
- 提示词保护与状态更新白名单。
- 停滞提示已从主线催促改为自由行动建议。

主要缺口：

- 自由行动还不稳定地产生信息、资源、关系、路线和后果。
- NPC 信息分层 UI 与普通 NPC prompt 已按“玩家已知 / 当前 NPC 私密 / DM 私密”收敛；仍需继续审查预设副本内容是否把秘密写进公开字段。
- 剧本仍容易用阶段主线牵引，地点/NPC/物品的可互动性没有成为默认体验。
- 成长和装备已能影响检定，但还没有稳定解锁新行动类型。

## 2. 总体目标

核心循环必须固定为：

```text
玩家自由表达 -> 系统理解意图 -> 风险/收益预览 -> 玩家确认或改写 -> 结算 -> 状态变化 -> 解锁新的行动可能
```

验收标准：

- 玩家连续 3-5 回合不做主线时，系统仍能给出合理叙事、状态收益或后果。
- 地点、NPC、物品、线索、关系至少有一种会因自由行动发生变化。
- 主线提示只能作为可选方向，不能成为停滞时的默认命令。
- 新手不需要理解“行动/计策/检定模式”，只需要描述想做什么。

## 3. 阶段路线

### 3.1 Phase A：自由行动闭环

目标：让非主线输入也能落到玩法资源上。

范围：

- 从当前地点、可达地点、NPC 公开信息、已知线索、背包工具生成行动建议。
- 让观察、询问、准备、交易、休息、使用物品、移动都能通过单输入框被识别。
- 普通自由行动可以产生低强度收益：线索提示、关系变化、风险下降、补给、时钟变化或支线入口。

不做：

- 不新增复杂战斗系统。
- 不引入新前端框架。
- 不让 AI 直接写入未校验状态。

验收：

- 推荐行动里至少 2 条来自当前场景上下文，而不是泛化模板。
- “我想自己试个办法……”能进入行动预览或普通叙事，而不是被强制收束到主线。
- 回归测试覆盖停滞提示和自由建议。

### 3.2 Phase B：NPC 信息分层

目标：NPC 是信息容器，而不是开局剧透卡。

数据层：

- `public`：姓名、外观、第一印象。
- `observed`：观察后可见的习惯、情绪、行为模式。
- `rumored`：听闻但未证实的信息。
- `confirmed`：证据支撑的信息。
- `private`：只供 GM/prompt 私密使用，永不直接展示。

验收：

- 角色详情页默认只展示玩家已知层级。
- 未解锁秘密不进入普通 UI。
- 通过观察、询问、证据、关系或检定可逐步解锁 NPC 档案。

### 3.3 Phase C：剧本网状结构

目标：副本从线性主线变为“主干 + 节点 + 多路线”。

要求：

- 每个预设副本至少 4 个地点节点。
- 每个主阶段 1-3 个挑战。
- 每个关键结论至少 3 条线索或替代路径。
- 至少 1 条社交路线、1 条调查路线、1 条资源/代价路线。
- 失败推进到新局面，而不是只阻断。

验收：

- 真实 API 通关记录中，玩家可以用不同路线抵达结局。
- 标准流程有 8-12 次有意义检定或等价代价。
- 自动推进主线不得连续超过 2 次玩家行动。

### 3.4 Phase D：成长、装备、探索整合

目标：资源影响玩法选择，而不仅是数值修正。

要求：

- 工具类装备解锁特定行动：扫描、维修、开锁、伪装、追踪。
- 消耗品提供一次性优势、风险下降或失败缓冲。
- 探索收益能降低后续 DC、开放替代路线或避免失败状态。
- 升级除属性点外，逐步引入专长或行动倾向。

验收：

- 至少 5 类物品能改变行动建议或行动预览。
- 至少 3 个探索奖励会影响后续挑战。
- 玩家准备充分时，能明显降低风险但不能跳过重大挑战。

### 3.5 Phase E：记录、回放与调试验收

目标：每次完整副本都能被审查。

要求：

- 通关/失败记录保留完整对话、关键状态变化、检定和结局。
- 回放文本清理协议标记和调试噪声。
- 测试脚本能模拟至少两个不同副本的真实 API 流程。

验收：

- 每次发布前至少完成：静态回归、真实 API 冒烟、线上文件检查。
- 测试记录可用于判断自由度、检定密度、NPC 信息泄露和主线催促问题。

## 4. 实施顺序

推荐顺序：

1. Phase A：自由行动建议上下文化。
2. Phase A：自由行动结算收益与后果。
3. Phase B：角色档案 UI 收敛为玩家已知信息。
4. Phase C：预设副本补足节点和多路线。
5. Phase D：装备/物品/探索奖励影响行动可能性。
6. Phase E：真实 API 多副本验收。

当前实现进度：

- 已完成自由行动建议上下文化：建议会读取当前地点、可达地点、线索动作、背包工具和 NPC 公开印象。
- 已完成自由行动轻量收益：低风险观察、询问、试探、调查和使用工具会写入玩家知识账本并记录事件，供右侧局势与后续 AI 叙事使用。
- 已完成自由行动线索阶段推进：玩家执行当前线索的可追查动作后，系统会推进 `clueGraph.currentStage`、记录“线索推进”事件并刷新下一步推荐；重复同一阶段动作不会刷进度。
- 已完成自由行动轻量后果处理：玩家明确道歉、解释、清理、稳定或修复时，可解除匹配的低/中严重度后果；高/关键后果仍要求检定、资源或正式挑战。
- 已完成 NPC 档案默认收敛：普通玩家视角只显示公开印象、已知线索和已解锁 hint；完整角色卡、私密钩子与真相只在作者/调试剧透开关下显示。
- 已完成 NPC prompt 信息边界收敛：普通 NPC/群聊 prompt 不再注入全局线索私密真相、隐藏失败条件、未知关键结论或完整剧情节拍；完整私密结构只进入 DM 叙事 prompt，当前 NPC 仅保留自身私密扮演资料。
- 已完成预设副本私密字段泄露门禁：回归测试会扫描三套预设模板的玩家可见字段，确保 NPC 动机、恐惧、秘密和筹码不会被原样写进开场、任务、地点、阶段、可见节点、提示和公开档案。
- 已完成支线/非主线目标进入局势推荐：未完成支线会按当前地点、在场 NPC、已知证据和知识相关性排序，作为可选方向出现；这只影响“看见和选择”，不会放松支线完成闸门。
- 已更新 Prompt 导演规则：玩家偏离当前节拍时应提供主线、支线、探索或代价选择，不再要求把玩家强制收束到主线。
- 已完成物品解锁行动建议：已装备工具、任务物品和可消耗资源会按当前挑战、地点和物品效果生成具体可输入行动，例如扫描、修复、开锁、伪装、追踪、封印、净化、路线规划或证据整理；未装备的非消耗工具仍先提示装备，避免误导常驻修正已经生效。
- 已完成基础商店工具覆盖：基础商店和单输入框购买命令支持伪装工具包、追踪工具包，准备建议会优先按当前挑战推荐具体工具，而不是只回退到通用工具。
- 已完成检定失败缓冲：玩家显式投入的 `clock_resist`、负向 `clock_delta` 或负向 `world_tension` 消耗品会在部分成功/失败/大失败后缓解时钟或世界紧张度；纯缓冲物成功时不消耗，未命中目标时也不消耗。
- 已完成探索证据绑定挑战：自由探索命中当前挑战方向时，生成的证据会绑定挑战、方法、核心结论和任务支持，进入右侧挑战证据链，并降低后续相关挑战行动的风险或 DC。
- 已完成探索路线解锁闭环：证据/发现把 `flowGraph.nodes` 路线节点变为可用后，会同步补齐地图 `connections`，推荐的“前往某地”可以被单输入框和地图移动真实执行。
- 已完成升级行动倾向：2 级起按最高属性自动派生轻量专长，进入准备提示和行动预览，匹配行动会降低风险/DC，但不会替代检定或证据链。
- 已完成失败风险缓解建议：公开时钟、可见反制或世界紧张度接近失败条件时，局势推荐会显示具体缓解行动；隐藏失败来源不会泄露。
- 已完成 AI 空回复诊断：API 成功但可见叙事为空时，不再写入空白角色/DM 消息；系统会显示 `【AI空回复】`、写入 `eventLog`，并累积 `scene.aiDiagnostics` 供复盘和排障。
- 已完成静态资源引用检查：回归测试会验证 `index.html` 的本地 JS/CSS/站内链接存在、不能逃出 GitHub Pages artifact，并要求本地 JS/CSS 缓存版本一致。
- 已完成 JS 语法发布门禁自动化：`tests/regression/all-js-syntax.test.js` 会在全量回归中统一执行所有项目 JS 和回归测试 JS 的 `node --check`。
- 已完成阶段回顾证据链强化：通关/失败记录会按挑战 id、任务支持和核心结论把检定、证据与阶段关联，避免关键证据只支持 revelation 时从阶段摘要中丢失。
- 已完成冒险回顾事件快照：通关/失败记录会保存 `eventLog` 的清理后快照，并在右侧回顾中折叠展示资源消耗、移动、探索、挑战、升级和结局等关键状态变化。
- 已完成历史尝试回看入口：右侧“记录与回顾”会展示 `runHistory` 中最近旧尝试的轻量卡片，让玩家比较多次通关/失败路线而不展开完整旧对话。
- 已完成同伴协助范围门禁：远程/承诺协助不再进入潜行、强突、战斗等需要现场参与的建议或检定资源，避免玩家调用暂时无关的角色解决物理行动。
- 已完成单一 DC 回归门禁：本地行动预览已有裁决时，AI 输出冲突 `[check:]` 会被统一到本地属性和 DC，避免玩家看到同一次行动的两个 DC。
- 已完成检定续写二次检定门禁：掷骰结果后的 DM 续写会清理并忽略新的 `[check:]`，但仍处理其它白名单标记，避免同一次行动结算后再次弹出检定卡。
- 已完成自动主线推进上限门禁：结构化副本中普通叙事自动完成主线目标会受 `maxAutoQuestAdvances` 限制，显式 `quest_update/questsUpdate` 没有证据、挑战或结论支持时仍会被拦截。
- 已完成全入口玩家协议标记拦截：普通输入、OOC/系统前缀输入都会在进入模型前经过 `PromptGuard.inspectUserInput()`，玩家夹带 `[event:]`、`[check:]` 或 `<state_update>` 不会进入对话流。
- 已完成工具行动路由闭环补强：追踪工具生成的“追踪痕迹”建议会被单输入框识别为观察型行动，进入物品修正和自由行动证据链；多个工具同时存在时，玩家明确点名的工具优先绑定，避免错记工具来源。
- 已完成普通输入检定上下文兜底：AI 对普通 `talk` 输入要求检定时，会读取消息保存的 `intentMeta.actionType`，即使不能重新规划行动，也能把扫描、修复等动作上下文传给检定物品和同伴资源匹配。
- 已完成同伴关键结论解锁回归门禁：`unlock.revelationIds` 默认要求对应结论 `confirmed`，不会因 `suspected` 提前公开同伴底牌；剧本显式写 `revelationStatus:"suspected"` 时才允许怀疑阶段解锁。
- 已完成预设副本网状结构门禁：回归测试会验证每个预设副本的地点连接存在、双向、从起点全图可达，且 `flowGraph` 节点唯一、每个地点能解析到节点、每个出口能解析到真实节点或地点，避免推荐移动指向死路。
- 已完成同伴资源真实角色绑定门禁：`companionResources` 即使写了 `immediate`、证据或关键结论解锁，也必须绑定到真实角色；缺少绑定或指向不存在角色时不会进入行动建议、检定资源或消耗流程，显式 `characterName` 仍可修复旧数据。
- 已完成状态补丁清洗防线补强：`PromptGuard.sanitizeStateUpdate()` 会清除 `api.key`、`api/key`、`localStorage/sessionStorage/indexedDB/cookie` 等敏感键变体和原型污染键；`StrategyManager.applyStateUpdate()` 入口也会二次清洗，防止未来调用绕过群聊提取入口。
- 已完成地点补丁地图连通性门禁：`locationUpdate.connections` 会先按真实地点集合过滤，丢弃不存在地点和自连接，再统一补齐双向出口；同一批新增地点可以被连接，推荐移动不会指向不可执行死路。
- 已完成自由行动压力缓解闭环：玩家明确稳定、压低、安抚、整理补给、修复或隔离时，可小幅降低一个公开/暗示时钟或世界紧张度；隐藏压力不会泄露，同一压力源同类意图会写入 `freeformReliefLog` 防止重复刷收益。
- 已完成自由行动 NPC 焦点归属修正：玩家在自然输入中明确点名某个在场 NPC 时，公开反应、关系收益和隐藏档案 hint 会归属到被点名角色；一句话点名多个 NPC 时不强行套用当前选中角色，避免误解锁或误改关系。
- 已完成自由行动反制缓解闭环：玩家自然输入命中可见反制的 counterplay、标题或提示时，会小幅削弱/揭示该反制并写入 `freeformCounterplayLog` 防重复；隐藏反制不会被自由行动探出或泄露。
- 已完成检定来源可见化：待掷骰卡会同时显示 `AI 要求检定` 或 `本地行动裁决` 以及对应行动意图，避免玩家只看到“来自行动”却不知道检定由谁提出。
- 已完成 NPC 在场状态可见化：角色档案默认展示结构化位置、地点和联系方式，让玩家理解某个 NPC 是否在场或能否远程联系；隐藏 agenda、离屏计划和自由文本 note 不进入普通玩家详情页。
- 已完成远程协助联系门禁：同伴资源已有明确 presence 时，`remote` 协助必须能远程联系才会进入建议、prompt 或检定卡；只知道 NPC 离场但没有联系方式时不会默认可用，旧存档未知 presence 仍按关系门槛兼容。

## 5. 发布门禁

每个阶段完成前必须满足：

- 新增或更新 SPEC。
- 有针对性回归测试。
- `node --check` 覆盖改动 JS。
- `tests/regression/*.test.js` 全量通过。
- 线上部署后确认核心静态文件包含新行为。
