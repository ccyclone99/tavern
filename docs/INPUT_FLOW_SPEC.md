# Tavern Natural Input Flow SPEC

本文档定义 Tavern 下一阶段输入与裁决体验。它覆盖“一个输入框完成所有功能”的目标，并替代 `docs/UIUX_SPEC.md` 中旧的“对话 / 行动 / 计策 / OOC 分段模式”作为后续实现方向。

## 1. 背景

最近的新手酒馆试玩暴露出几个一致问题：

- 玩家不知道什么时候该“切行动”“切计策”。
- 玩家把“检定”理解成需要主动选择的功能，而不是 DM 在必要时触发的裁决。
- 同一次行动可能出现两个 DC，例如 AI 说 DC10，本地行动预览又给 DC16。
- 玩家问“什么情况”“然后呢”“我要怎么操作”时，系统继续推进剧情，而不是解释当前操作状态。
- 系统消息、离屏行动、关系变化和行动意图混在聊天流里，干扰玩家阅读剧情。

结论：默认体验不应让玩家操作模式面板。玩家只需要描述自己想说什么、做什么或计划什么。系统负责识别意图、提示风险、触发检定、更新状态。

## 2. 目标

### 2.1 玩家只面对一个主输入框

玩家的主要操作入口只有 `chatInput`：

```text
你想说什么，或打算怎么做？
```

玩家可以自然输入：

- `老板，来杯牛奶。`
- `@艾莉 你觉得我该怎么偷走那块派？`
- `我观察吧台附近有没有人盯着我。`
- `我趁莫里斯转身找青柠的时候，把苹果派藏进袖子里。`
- `我想制定一个偷派嫁祸给艾莉的计划。`
- `怎么搞？`
- `掷骰。`

这些输入都应通过同一个输入框处理。

### 2.2 检定是被动触发

检定不是玩家主动选择的模式。玩家主动选择的是行动，系统在满足以下条件时触发检定：

- 行动结果不确定。
- 行动有失败代价。
- 行动会改变局势、关系、资源、线索、地图或时钟。
- NPC、环境、敌方反制或隐藏风险正在抵抗玩家目标。

玩家可以输入“掷骰”来响应待处理检定，但不能通过“我要检定”直接让世界产生一次没有行动上下文的裁决。

### 2.3 输入框始终解释当前状态

玩家不应需要回读多轮聊天才能知道下一步。

输入区必须在以下状态给出明确短提示：

- idle：等待玩家说话或行动。
- pending_action：已生成行动预览，等待执行、取消或改写。
- pending_check：已要求检定，等待掷骰或取消。
- help_context：玩家正在询问规则或操作，先解释当前状态。
- blocked_streaming：AI 回复中，不能重复发送。

### 2.4 裁决来源唯一

同一次行动只能有一个有效的属性和 DC。

推荐规则：

- 本地规则层 `ActionPlanner` / 后续 `IntentRouter` 是行动风险、建议属性和 DC 的权威来源。
- AI 负责叙事、提出代价、解释局势和根据检定结果续写。
- 当存在 `pendingAction.suggestedCheck` 时，AI 不得生成冲突 DC；即使输出了不同 `[check:]`，客户端也必须以 `pendingAction` 中的裁决为准，或显示“裁决已按本地规则统一”。
- 当没有 `pendingAction` 时，AI 可以提出 `[check:属性|DC]`，但客户端仍需记录来源为 `ai`，并在 UI 上说明。

### 2.5 系统降噪

聊天流优先展示玩家能理解的剧情文本。以下内容默认不直接插入聊天流，或以折叠摘要显示：

- NPC 离屏行动。
- 关系微小变化。
- 行动预览内部字段。
- 局势时钟的低级别推进。
- 规则层调试文本。

这些内容应进入右侧“局势”“任务”“线索”“计策”或“日志”区域。

## 3. 非目标

本阶段不做：

- 新增复杂战斗系统。
- 引入 React/Vue 或构建系统。
- 删除现有 `pendingAction` / `pendingCheck` 数据结构。
- 让 AI 自动替玩家确认高风险行动。
- 将 NPC 私密信息直接公开给玩家。
- 取消高级快捷入口。快捷入口可以保留，但不能成为默认必经路径。

## 4. 核心交互模型

### 4.1 玩家心智模型

玩家只需要理解四句话：

1. 想说什么就直接说。
2. 想做什么就直接描述动作。
3. 有风险时系统会先提醒你。
4. 需要骰子时系统会让你掷。

不要求新玩家理解：

- 输入模式。
- `/strategy`。
- `action_intent`。
- `[check:]`。
- pending 状态字段。

### 4.2 默认输入区

idle 状态：

```text
Placeholder: 你想说什么，或打算怎么做？
Hint: 可以直接聊天、观察、询问、行动或制定计划。高风险行动会先让你确认。
Button: 发送
```

pending action 状态：

```text
Placeholder: 输入“执行”确认，输入“取消”放弃，或直接改写行动。
Hint: 这一步还没有进入剧情；确认后才会推进。
Button: 执行
```

pending check 状态：

```text
Placeholder: 输入“掷骰”继续，或输入“取消”放弃检定。
Hint: 检定来自刚才的行动，掷骰后会继续叙事。
Button: 掷骰
```

help context 状态：

```text
Placeholder: 继续问规则，或直接输入你想做的事。
Hint: 当前状态会用一句话解释，不推进剧情。
Button: 发送
```

### 4.3 快捷 chips

输入框上方或下方可以显示 2 到 4 个上下文建议。它们是辅助，不是必须点击。

当前实现已接入基础 chips：

- idle：观察四周、询问当前对象或在场 NPC、制定计划、我该做什么。
- pending_action：执行、取消、说明风险、改写行动。
- pending_check：掷骰、取消检定、解释检定。

示例：

- `观察四周`
- `问艾莉建议`
- `执行这个动作`
- `掷骰`
- `取消`
- `制定计划`

点击 chip 的行为：

- 对普通建议：填入输入框，允许玩家修改。
- 对执行/掷骰/取消：直接调用对应 pending 处理，但必须有清晰状态变化。
- 对高风险建议：先生成行动预览，不直接结算。

## 5. Intent Router

需要新增或抽象一个输入路由层，建议命名为 `IntentRouter`。它位于 `ChatUI.onSend()` 和 `GroupChat.handleUserMessage()` 之间。

### 5.1 输入处理顺序

每次发送文本按以下优先级处理：

| 优先级 | 条件 | 行为 |
|---|---|---|
| 1 | `State.isStreaming` | 阻止重复发送，提示等待或停止 |
| 2 | `scene.pendingCheck` 存在 | 优先处理掷骰、取消、帮助，不把普通文本送进剧情 |
| 3 | `scene.pendingAction` 存在 | 处理执行、取消、改写行动 |
| 4 | 明确 OOC/帮助 | 生成 UI 帮助或 OOC 消息，不推进世界时钟 |
| 5 | 明确指定对象 | 写入 target/participants，并按目标 NPC 路由 |
| 6 | 明确计策/计划 | 创建或推进 strategy intent |
| 7 | 有风险动作 | 生成行动预览或直接进入行动结算 |
| 8 | 观察/询问/轻量互动 | 作为普通 RP 或低风险行动发送 |
| 9 | 默认闲聊 | 普通对话 |

### 5.2 pendingCheck 状态

当 `scene.pendingCheck` 存在时：

- `掷骰`、`roll`、`开始`、`继续`、空格回车等明确继续意图：调用 `GroupChat.rollPendingCheck()`。
- `取消`、`算了`、`不做了`：调用 `GroupChat.cancelPendingCheck()`。
- `这是什么`、`怎么搞`、`我要怎么操作`：显示帮助，不推进剧情。
- 其他文本：提示“当前先完成检定；输入‘掷骰’继续，或‘取消’放弃。”不发送给 AI。

pendingCheck 不允许被普通 RP 文本绕过，否则会造成检定结果悬空和世界时钟重复推进。

### 5.3 pendingAction 状态

当 `scene.pendingAction` 存在时：

- `执行`、`确认`、`开始`、`就这样`：调用 `ChatUI.confirmPendingAction()`。
- `取消`、`算了`、`不做了`：调用 `ChatUI.cancelPendingAction()`。
- `改成...`、`换成...`、`不是，我想...`：用新文本重建 `pendingAction`。
- `为什么这么高风险`：解释风险来源，不推进剧情。
- 其他明确动作文本：默认替换当前 preview，而不是直接进入剧情。

### 5.4 帮助/OOC 识别

以下输入应优先作为帮助或 OOC，而不是 NPC 台词：

- `怎么搞`
- `什么情况`
- `我要怎么操作`
- `这个按钮什么意思`
- `经验值干嘛用`
- `检定是什么`
- `我现在该干什么`
- `怎么保存`

帮助回复可以是 UI 级提示，也可以插入 `type: "system"` 的短消息。它不应触发：

- NPC 离屏行动。
- 关系变化。
- 世界时钟推进。
- AI 主叙事续写。

### 5.5 目标角色路由

玩家可以通过以下方式指定对象：

1. 文本 `@角色名`。
2. 当前选中的角色。
3. 上一轮直接互动对象。
4. 教程当前导师。
5. 群聊默认发言规则。

路由优先级：

```text
@mention > selectedCharacterId > pending context target > last addressed NPC > tutorial mentor > group
```

如果玩家问的是系统操作，不应强行分配给 NPC。若需要角色内解释，也应由当前导师解释，并避免推进剧情。

## 6. 行动与检定裁决

### 6.1 行动分类

行动类型保留现有枚举：

- talk
- observe
- ask
- probe
- lie
- threaten
- persuade
- investigate
- sneak
- force
- combat
- trade
- use_item
- rest
- strategy

单输入框还承接本地系统命令，不进入 AI 叙事，包括移动、装备/卸下、使用物品、购买基础补给、出售非关键物品、休息和分配属性点（如“加一点敏捷”“体质+1”）。

新增路由含义：

- 玩家不需要手动选择类型。
- `IntentRouter` 根据文本、上下文、目标 NPC、当前地点和 pending 状态自动分类。
- 分类结果写入消息或 pendingAction 的 `intentMeta`，便于调试和回放。

建议结构：

```js
{
  kind: "action",
  actionType: "sneak",
  confidence: 0.82,
  targetCharacterIds: ["char_morris"],
  targetLocationId: "tavern_bar",
  isRisky: true,
  needsPreview: true,
  reason: "偷取物品且目标 NPC 在场"
}
```

### 6.2 行动预览触发

必须先预览：

- 暴力、偷窃、潜行、欺骗、威胁、战斗。
- 会消耗物品、金币、生命、时间或机会。
- 会明显影响 NPC 关系。
- 会暴露计策或触发反制。
- 玩家描述不够具体，但行动有高风险。

可以不预览：

- 闲聊。
- 低风险提问。
- 观察公开环境。
- 明确安全的移动。
- 教程中没有失败代价的练习动作。

模糊情况：

- 使用 inline 提问，而不是直接推进。
- 例如：“你是想开玩笑地吓唬他，还是认真威胁他？”

### 6.3 单一 DC 规则

当 `pendingAction.suggestedCheck` 存在时：

```js
pendingAction.adjudication = {
  source: "local",
  stat: "dexterity",
  statName: "敏捷",
  dc: 16,
  reason: "偷取物品，目标 NPC 在附近，存在敌方反制",
  risk: 69
}
```

后续检定卡必须使用该裁决。

AI 回复中如果输出：

```text
[check:敏捷|DC10]
```

但当前 action adjudication 是 `敏捷 DC16`，客户端应：

- 使用 DC16。
- 在开发日志中记录冲突。
- 不向玩家展示两个 DC。
- 给 AI 的后续 prompt 注入统一后的检定结果。

### 6.4 AI check 标记

推荐将 AI 检定标记扩展为：

```text
[check:auto]
```

含义：

- 若存在最新 `pendingAction.adjudication`，使用本地裁决。
- 若不存在，则 AI 必须提供传统 `[check:属性|DC]`。

兼容旧格式：

```text
[check:敏捷|DC16]
```

### 6.5 掷骰结果

掷骰后写入 `type: "check"` 消息，但展示为检定结果卡，而不是普通玩家台词。

结果必须包含：

- D20 点数。
- 属性修正。
- 物品修正。
- DC。
- 结果等级。
- 后果提示。
- 使用的行动上下文。

结果等级沿用：

- critical_success
- success
- partial
- fail
- critical_fail

### 6.6 失败推进

失败不能只说“失败”。每次失败至少产生一种推进：

- NPC 发现但提出条件。
- 关系、警觉或时钟变化。
- 玩家得到不完整信息。
- 目标转移，形成新机会。
- 暴露计策，但开启反制玩法。
- 资源损失，但剧情继续。

检定前 UI 应展示失败方向，而不是只展示 DC。

## 7. 计策输入

计策不再依赖“计策模式”。触发方式：

- 玩家说“我想制定一个计划...”
- 玩家说“我的计策是...”
- 玩家提出多阶段目标。
- 玩家从右侧计策面板点击“推进此计策”，输入框获得上下文。
- AI/DM 追问“你打算分几步做？”

### 7.1 计策创建

创建计策时，系统应收集：

- 目标。
- 目标对象。
- 已知情报。
- 可用资源。
- 第一步行动。
- 暴露风险。

不要求玩家一次填完整。缺少信息时由 DM 追问。

### 7.2 计策与行动关系

计策是长期容器，行动是具体步骤。

示例：

```text
计策：偷派嫁祸给艾莉。
步骤 1：趁莫里斯转身偷走苹果派。
步骤 2：制造艾莉接触过盘子的迹象。
步骤 3：引导莫里斯怀疑艾莉。
```

每一步都可以独立生成行动预览和检定。

### 7.3 计策状态展示

计策更新应主要展示在右侧“计策”面板。聊天流只显示自然叙事和少量关键摘要。

## 8. 系统消息降噪

### 8.1 消息分层

| 类型 | 默认位置 | 聊天流展示 |
|---|---|---|
| 剧情叙事 | 聊天流 | 展示 |
| 玩家台词/动作 | 聊天流 | 展示 |
| 检定结果 | 聊天流卡片 | 展示 |
| 关键任务变化 | 聊天流短摘要 + 右侧任务 | 展示摘要 |
| 新线索 | 右侧线索 + 角标 | 可选短提示 |
| 关系微调 | 右侧详情/日志 | 默认折叠 |
| NPC 离屏行动 | 右侧局势/日志 | 默认不展示 |
| 行动预览内部文本 | 输入区卡片 | 不写入聊天历史 |
| 调试/协议文本 | console/dev | 不展示 |

### 8.2 关系变化

关系变化应降低频率。

建议：

- 普通教学鼓励不刷好感。
- 轻微玩笑不每次更新关系。
- 只有明确选择、风险行动、关键剧情、礼物、伤害、背叛、帮助等事件才展示关系变化。
- 小幅变化可累计后摘要，例如“艾莉对你的看法略有改善”。

### 8.3 离屏行动

NPC 离屏行动进入局势日志。聊天流只在以下情况显示：

- 离屏行动已影响当前场景。
- 玩家发现了行动痕迹。
- 离屏行动触发公开事件。
- 它是当前回合最重要的变化。

## 9. 教程重写要求

新手教程应避免让玩家学习 UI 模式，改为学习自然行动。

### 9.1 教程步骤

旧步骤：

```text
对话 -> 指定角色 -> 计策模式 -> D20 检定 -> 地图移动
```

新步骤：

```text
直接说话 -> 指定对象 -> 描述行动 -> 系统要求掷骰 -> 查看后果 -> 地图移动
```

### 9.2 话术约束

禁止把检定说成玩家要主动打开的功能：

```text
错误：你选择“检定”模式。
正确：你描述一个有风险的动作，我会在需要时让你掷骰。
```

禁止要求玩家切到计策模式：

```text
错误：把输入区切到“计策”模式。
正确：直接输入“我想制定一个计划：...”。
```

### 9.3 卡住时处理

玩家输入：

```text
怎么搞？
什么情况？
然后呢？
我要怎么操作？
```

教程应返回操作提示，而不是继续推进剧情。

示例：

```text
当前：莫里斯在等你描述偷派动作。
你可以直接输入：“我趁他转身时，把苹果派藏进袖子里。”
```

## 10. UI 规格

### 10.1 顶部模式按钮

旧的“对话 / 行动 / 计策 / OOC”按钮默认隐藏，只保留代码兼容能力：

- 不作为主流程必需控件。
- 不出现在默认输入区视觉层级。
- 不要求教程点击。
- 不改变玩家对自然输入的理解。

可选替代：

- 隐藏到输入框左侧的工具菜单。
- 改为 chips：“OOC”“制定计划”“高风险行动”。
- 作为调试/高级模式开关。

### 10.2 输入区提示

输入区必须显示当前状态提示：

- idle：`直接输入你想说的话或动作。`
- risky draft：`这看起来有风险，我会先让你确认。`
- pending action：`输入“执行”确认，或直接改写。`
- pending check：`输入“掷骰”继续。`
- help：`当前问题不会推进剧情。`

### 10.3 行动预览卡

行动预览卡应更像“DM 在确认风险”，而不是表单。

必须显示：

- 你要做什么。
- 为什么有风险。
- 可能用哪个属性。
- 失败会怎样推进。
- `执行` / `取消` / `改写`。

不应显示过多内部字段，例如完整 prompt 文本。

### 10.4 检定卡

检定卡应明确来源：

```text
来自行动：偷走苹果派
敏捷 DC16
失败：莫里斯发现你，但可能提出赔礼或交换条件。
```

按钮：

- `掷骰`
- `取消`

输入框也必须能处理“掷骰”和“取消”。

## 11. 数据结构建议

现有结构可继续使用：

- `scene.pendingAction`
- `scene.pendingCheck`
- `State.inputMode`
- `message.type`
- `message.visibility`

## 12. Prompt 约束

### 12.1 DM prompt

PromptBuilder 应加入约束：

```text
玩家不需要主动选择检定。只有当玩家描述了有风险且结果不确定的行动时，你才提出检定。
若系统已提供建议检定，请沿用该属性和 DC，不要另定 DC。
如果玩家问“怎么操作”“什么情况”，先解释当前状态，不推进剧情。
```

### 12.2 NPC prompt

NPC 不应抢答系统帮助问题，除非它是教程导师。

对普通世界：

- 操作问题由系统或 DM 回答。
- 角色内规则解释应短，避免推进剧情。

对教程世界：

- 导师可以角色内解释，但必须明确下一步输入示例。

## 14. 验收场景

### 14.1 基础帮助

输入：

```text
怎么搞？
```

期望：

- 系统解释当前可做什么。
- 不推进时钟。
- 不触发 NPC 离屏行动。
- 不改变关系。

### 14.2 检定概念

输入：

```text
检定
```

期望：

- 系统解释“检定会在有风险行动时触发”。
- 提示玩家描述一个动作。
- 不创建无上下文检定。

### 14.3 偷派行动

输入：

```text
我趁莫里斯转身找青柠的时候，悄悄把苹果派藏进袖子里。
```

期望：

- 系统识别为 sneak/use_item 或 sneak。
- 若风险高，先显示行动预览。
- 只展示一个建议属性和 DC。
- 玩家输入“执行”后进入 DM 结算。

### 14.4 掷骰

前置：存在 `scene.pendingCheck`。

输入：

```text
掷骰
```

期望：

- 直接调用 `rollPendingCheck()`。
- 插入检定结果卡。
- AI 根据结果续写。
- 世界回合只推进一次。

### 14.5 改写行动

前置：存在 `scene.pendingAction`。

输入：

```text
改成我先夸莫里斯的手艺，再请他送一小块派。
```

期望：

- 替换 pendingAction。
- 不推进剧情。
- 新预览风险和 DC 重新计算。

### 14.6 计策自然触发

输入：

```text
我想制定一个计划，偷走苹果派然后嫁祸给艾莉。
```

期望：

- 创建或推进计策。
- AI/DM 追问第一步或可用资源。
- 不要求玩家切换模式。

### 14.7 系统降噪

玩家完成一次普通对话。

期望：

- 聊天流不刷出多条离屏行动。
- 若有关系微调，默认折叠或进入右侧日志。
- 状态摘要最多显示一条关键变化。
