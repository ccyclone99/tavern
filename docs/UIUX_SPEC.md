# Tavern UI/UX SPEC

本文档定义 Tavern 下一阶段前端 UI/UX 优化目标。它不改变 `docs/SPEC.md` 中的游戏规则，而是约束这些规则如何被玩家理解、操作和验证。

## 1. 目标

当前版本已经具备叙事 RPG 的核心规则：行动预览、检定卡、知识边界、局势时钟、NPC 离屏行动、计策和物品效果。下一阶段 UI/UX 的目标是让玩家在不读文档的情况下理解：

- 我现在在哪里。
- 我当前最重要的目标是什么。
- 我知道了什么，还不知道什么。
- 我可以说话、行动、制定计策或调查线索。
- 一次行动为什么有风险，结果影响了哪些状态。
- NPC 信息为什么没有一开始公开，以及哪些信息已被逐步解锁。

核心衡量标准：玩家每次回到页面后，10 秒内能判断当前局势，30 秒内能选择下一步行动。

## 2. 非目标

当前 UI/UX 阶段不做：

- 引入 React/Vue/构建系统。
- 重做为落地页或宣传页。
- 大规模替换 IndexedDB 数据结构。
- 重新设计核心规则。
- 将 NPC 私密设定直接公开给玩家。
- 增加联网后端、账号系统或多人协作。

## 3. 设计原则

### 3.1 游戏优先，不是聊天框优先

聊天仍是主要叙事界面，但第一屏必须表达这是一个可博弈的叙事 RPG，而不是普通 AI 对话框。

优先露出：

- 当前局势。
- 主要目标。
- 玩家资源。
- 行动入口。
- 最近后果。

普通闲聊、行动、计策、调查应通过同一个输入框自然表达。系统负责识别玩家意图，并在必要时展示行动预览、检定卡或帮助提示。旧的输入模式按钮默认隐藏，只保留代码兼容，不能成为新手流程的必经步骤。输入流细则见 `docs/INPUT_FLOW_SPEC.md`。

### 3.2 信息分层

玩家习惯上会寻找“已知事实”和“未解锁内容”的边界。UI 必须清楚区分：

- 已确认事实。
- 传闻或推测。
- 未解锁档案槽。
- GM/NPC 私密信息。

未解锁内容可以显示槽位、条件暗示或问号，但不得泄露真实文本。

### 3.3 行动要有预期

高风险行为在确认前必须让玩家看见：

- 行动类型。
- 建议属性。
- 风险等级。
- 关键修正来源。
- 失败推进方向。

确认后再进入 AI 结算。取消行动不应写入历史，也不应推进回合。

### 3.4 后果要可追踪

一次 AI 回复、检定或计策结算后，UI 应能让玩家追踪“发生了什么变化”：

- 关系变化。
- 线索新增或可信度变化。
- 时钟推进。
- 任务目标变化。
- 物品获得、消耗或可用。
- 反制出现、暴露或解决。
- 最近事件日志：检定、奖励、购买、移动、升级等关键变化可在局势面板回看。

这些变化不必全部弹窗，但必须在右侧面板、消息卡或状态摘要中可见。

### 3.5 操作密度适中

Tavern 是长期游玩的工具型界面，不应做成营销式大卡片页面。界面应保持安静、可扫描、信息密度适中。

卡片用于 repeated items、检定、行动预览、模态窗口；页面区域不应层层套卡。

## 4. 当前主要 UX 问题

### 4.1 右侧栏 tab 过多

当前右侧栏包含：局势、计策、线索、世界书、地图、任务、背包、详情。玩家会频繁在任务、局势、线索、计策之间切换，认知负担偏高。

优化方向：

- 将高频 tab 排在前 4 个：局势、任务、线索、计策。
- 将低频或编辑性质入口降级：世界书、详情、设置类入口。
- 给有新变化的 tab 使用非侵入角标。
- 右侧默认 tab 应根据上下文切换：新检定后偏向局势，新增线索后偏向线索，计策更新后偏向计策。

### 4.2 输入流不应要求玩家切模式

基线版本中存在“对话 / 行动 / 计策 / OOC”分段按钮。它解决了一部分可见性问题，但试玩显示新玩家仍会把“检定”“计策”“行动”理解成需要主动点击的功能，从而在剧情中卡住。

下一阶段优化方向：

- 默认只依赖一个主输入框。
- 玩家自然输入说话、观察、询问、行动、计划或求助。
- 系统通过 `IntentRouter` 自动识别意图。
- 高风险行动自动生成风险预览，确认前不推进剧情。
- 检定由系统在有风险且结果不确定时触发，不作为玩家主动模式。
- 计策通过“我想制定计划...”等自然语言触发，不要求切换计策模式。
- OOC/帮助问题自动识别，例如“怎么搞”“什么情况”“我要怎么操作”。

### 4.3 行动/检定反馈还不够“桌游”

行动预览和检定卡已经有基础版，但还需要更强的可解释性。

优化方向：

- 将风险来源分为正向修正、负向修正、未知风险。
- 检定卡显示 D20、属性、自动物品、可用消耗品、DC 和结果等级。
- 检定结束后的历史卡保留后果提示和状态变更入口。
- 消耗品和同伴协助必须由玩家显式点选；只有选中后掷骰才扣除 uses/数量。

### 4.4 玩家回流缺少“继续玩”入口

刷新或隔天回来后，玩家需要重新读聊天记录才能知道现在要做什么。

优化方向：

- 第一视线区域显示当前地点、主目标、最近风险和推荐行动。
- 右侧“局势”成为默认回流面板。
- 顶部栏保留场景名，但不要只显示场景名；应辅助显示当前地点或压力状态。

### 4.5 NPC 档案容易误导

玩家会自然点击角色查看“真相”。但本项目设计要求 NPC 信息逐步解锁。

优化方向：

- 角色详情默认展示玩家档案，不展示完整角色卡。
- 已知特征、传闻、关系、把柄、未解锁槽位分区。
- 编辑/剧透入口必须视觉上与玩家档案分离。
- 已解锁信息应标注来源：观察、对话、检定、线索、剧情事件。

### 4.6 移动端侧边栏占用高

移动端左右侧栏目前是覆盖式，功能可用，但高频信息不够轻。

优化方向：

- 移动端底部优先保留输入和核心状态。
- 左右侧栏打开时必须有清晰关闭路径。
- 右侧 tab 在移动端可横向滚动，但目标触控区域不小于 44px。
- 行动预览和检定卡不能把输入区完全挤出视口。

## 5. 信息架构

### 5.1 顶部栏

顶部栏负责全局定位，不承载复杂操作。

必须显示：

- 场景名。
- 当前地点或当前局势摘要。
- 左侧角色入口。
- 右侧情报/局势入口。
- 设置入口。

建议显示：

- 时钟压力简短状态。
- 最近一次重要状态变化提示。

不得显示：

- NPC 私密事实。
- 长文本任务描述。

### 5.2 左侧栏：角色与参与者

左侧栏用于选择当前互动对象和查看可见角色。

必须支持：

- 当前选中角色高亮。
- 在场/离场/未知状态。
- 关系或态度的轻量提示。
- 新解锁档案角标。

不应在默认卡片中展示：

- 真实动机。
- 秘密。
- GM-only 备注。

### 5.3 中央区：叙事和结算

中央区是叙事历史和当前操作的主舞台。

消息类型必须视觉区分：

- 玩家对话。
- 玩家行动意图。
- AI/NPC 回复。
- 系统事件。
- 检定结果。
- 计策状态变化。
- 离屏行动提示。

交互要求：

- 流式回复期间显示明确状态。
- 停止生成后不推进回合。
- 检定 pending 时输入区应提示先掷骰或取消。
- 结算卡不应与普通对话混在一起无法扫描。

### 5.4 右侧栏：玩家可行动信息

右侧栏不是资料仓库，而是“下一步决策面板”。

推荐 tab 顺序：

1. 局势。
2. 任务。
3. 线索。
4. 计策。
5. 背包。
6. 地图。
7. 角色档案。
8. 世界书/编辑。

右侧栏每个 tab 都应有明确空状态，而不是只显示“暂无”。

### 5.5 输入区

输入区是玩家表达意图的地方，而不是选择功能模式的地方。

默认能力：

- 普通 RP 对话。
- 指定角色，例如 `@艾莉`。
- 观察、询问、调查、移动等轻量行动。
- 偷窃、威胁、战斗、欺骗等高风险行动。
- 多阶段计策。
- OOC/帮助问题。
- pending action 的执行、取消和改写。
- pending check 的掷骰和取消。

输入区必须根据状态改变：

- placeholder。
- hint 文案。
- 主按钮文案。
- 建议 chips。
- pending preview/check 的展示逻辑。

旧的模式按钮默认隐藏，仅保留代码兼容；不得成为默认教程和核心流程的必需操作。

## 6. 组件规格

### 6.1 状态摘要条

位置：顶部栏或聊天区上方。

内容：

- 当前地点。
- 主目标一句话。
- 最高优先级时钟或风险。
- 最近变化。

验收：

- 不超过两行。
- 移动端不遮挡输入。
- 点击可打开右侧“局势”。

### 6.2 行动预览卡

触发：玩家自然输入了有风险、不可逆或会显著改变局势的行动后。

内容：

- 行动意图。
- 行动类型。
- 风险等级。
- 建议检定。
- 正向修正。
- 负向修正。
- 失败推进。
- 确认/取消。

验收：

- 取消不写入消息、不推进时钟。
- 确认后写入 `action_intent`。
- 风险条颜色符合语义：低风险、中风险、高风险、极高风险。
- 玩家输入“执行”“确认”“取消”“改成...”可以直接处理 pending action。

### 6.3 检定卡

触发：本地规则或 AI 在有行动上下文时要求检定。

内容：

- 属性与 DC。
- D20 公式。
- 属性修正。
- 自动物品修正。
- 可点选消耗品和同伴协助；选中后即时显示修正变化，掷骰时才消耗。
- 风险/赌注。
- 掷骰/取消。

验收：

- pending check 存在时不会重复推进回合。
- 掷骰结果进入历史卡。
- 未点选的消耗品不扣除；点选后掷骰会扣除对应 uses/数量。
- 历史卡保留结果等级和后果提示。
- 玩家输入“掷骰”或点击按钮都能完成检定。
- 同一次行动只出现一个有效 DC。

### 6.4 局势面板

内容：

- 当前地点。
- 主任务下一目标。
- 公开时钟。
- 模糊隐藏压力。
- 可见反制。
- 最近风险。
- 推荐行动。
- 可用线索。

验收：

- 玩家刷新回来后默认能看懂下一步。
- 时钟和反制有进度表现。
- 推荐行动点击后能填入输入区，或在明确安全时直接触发对应 pending 流程。

### 6.5 线索账本

内容分类：

- 传闻。
- 已确认线索。
- 证据。
- 推论。
- 真相。
- 未解锁问题。

筛选维度：

- 角色。
- 地点。
- 势力。
- 任务。
- 计策。

验收：

- 只读 `scene.knowledge` 和已解锁档案。
- 不读取 NPC 私密字段作为玩家可见文本。
- 新线索出现后右侧 tab 有角标。

### 6.6 角色档案

默认视图：

- 名称、头像、公开身份。
- 当前关系。
- 已观察特征。
- 已知传闻。
- 已确认情报。
- 欠债/把柄。
- 未解锁槽位。

编辑/剧透视图：

- 必须明确标注为编辑或剧透。
- 不作为默认玩家视图。

验收：

- 玩家不能通过默认详情页看到秘密、真实动机、隐藏 agenda。
- 解锁新档案后能看到来源和时间。

### 6.7 背包与物品效果

内容：

- 物品类型。
- 数量/uses。
- tags。
- effects 的玩家可读解释。
- 是否装备。
- 是否可用于当前检定或计策。

验收：

- 装备与非消耗任务物品可以进入自动修正。
- 消耗品必须显式选择后才扣除。
- 当前稳定版应提供显式选择 UI：资源只在玩家点选并掷骰后消耗。

## 7. 视觉系统

### 7.1 色彩

保留酒馆暗色基调，但避免所有界面都读成棕金单色。

语义色：

- 金色：主行动、当前选择、重要资源。
- 绿色：成功、恢复、优势。
- 琥珀：警告、部分成功、可用资源。
- 红色：危险、失败、伤害。
- 蓝色：知识、线索、系统信息。

要求：

- 同一屏不应只依赖金色表达所有重点。
- 风险、成功、线索、系统状态必须有可区分颜色和文本标签。

### 7.2 排版

原则：

- 聊天正文重可读性。
- 控件文字重扫描。
- 面板标题短而明确。

要求：

- 不使用负 letter-spacing。
- 不按 viewport 宽度缩放字体。
- 按容器尺寸控制文字换行。
- 紧凑面板内不用 hero 级字号。

### 7.3 卡片与控件

要求：

- 卡片圆角不超过 8px。
- 避免卡片套卡片。
- 按钮高度桌面不小于 36px，移动端不小于 44px。
- 工具类按钮优先使用图标，保留 hover/title 或可访问名称。
- 文本按钮只用于明确命令。

### 7.4 动效

允许：

- 新消息淡入。
- tab 角标轻微提示。
- 掷骰动画。
- 风险条变化。

不允许：

- 大面积装饰性光球、渐变球、模糊背景球。
- 影响阅读的持续动画。
- 导致布局跳动的 hover 状态。

## 8. 响应式规格

### 8.1 桌面端

目标宽度：1280px 及以上。

布局：

- 左侧角色栏固定宽度。
- 中央叙事区弹性扩展。
- 右侧决策栏固定宽度或可折叠。
- 输入区固定在底部。

验收：

- 消息、右侧 tab、输入区互不遮挡。
- pending action/check 不挤压到无法输入。
- 右侧高频 tab 不换行成难读状态。

### 8.2 平板端

目标宽度：768px - 1279px。

布局：

- 左右侧栏可覆盖或折叠。
- 中央区保持完整输入体验。
- 右侧栏打开时遮罩可关闭。

验收：

- 触控目标不小于 44px。
- 横屏和竖屏都能打开右侧 tab。

### 8.3 手机端

目标宽度：360px - 767px。

布局：

- 输入区始终优先可达。
- 侧栏全屏覆盖。
- tab 横向滚动。
- 局势摘要压缩为一行或两行。

验收：

- 不出现水平页面滚动。
- 长角色名、长任务名、长物品名不溢出容器。
- 检定卡出现后仍能看到掷骰按钮。

## 9. 可访问性

要求：

- 所有按钮有可访问名称。
- 键盘可聚焦主要操作。
- `:focus-visible` 清楚可见。
- 色彩语义不能只依赖颜色，必须有文本或图标辅助。
- 动画不影响阅读。
- 输入框、按钮、tab 的状态变化应同步 class 和 ARIA 状态。

最低验收：

- 仅键盘可完成：发送消息、处理建议 chips、确认/取消行动、掷骰/取消检定、打开右侧 tab。
- 屏幕阅读器能读出主要按钮作用。

## 10. 性能与稳定性

要求：

- 保持纯静态部署。
- 不引入重型前端框架。
- 不新增阻塞首屏的大型依赖。
- DOM 渲染保持增量更新，避免每条消息都全量重绘。
- 右侧列表长内容需要裁剪或分页。

验收：

- 本地页面无控制台 error。
- 100 条消息场景下滚动仍可用。
- 移动端打开/关闭侧栏无明显卡顿。

## 11. 实施阶段

### Phase 1: 信息架构与导航

- 调整右侧 tab 顺序。
- 增加局势摘要条。
- 统一空状态。
- 将角色详情默认改为玩家档案视角。

验收：

- 玩家刷新回来后能直接理解当前地点、目标、风险。
- 默认 UI 不泄露 NPC 私密信息。

### Phase 2: 自然输入与行动链路

- 将默认流程改为单输入框自然输入。
- 增加 `IntentRouter` 或等价输入路由。
- 强化行动预览卡。
- 强化检定 pending 和历史结果卡。
- 推荐行动可一键填入输入区。
- 模式按钮默认隐藏，不作为新手必经路径。

验收：

- 玩家不切模式也能完成对话、行动、计策和检定。
- OOC/帮助问题不会推进剧情。
- 行动确认前不推进剧情。
- 检定结算只推进一次回合。
- 同一次行动只出现一个有效 DC。

### Phase 3: 右侧决策面板

- 局势、任务、线索、计策形成联动。
- 新线索、任务变化、反制变化有角标。
- 背包显示物品 effects 的玩家可读解释。

验收：
